# `Binding.resolve/2` Consults BEAM First, Source AST Second

## Context

Red-team 04a#h8 flagged a correctness gap: a source-AST-first resolver
cannot see functions generated by `use SomeDSL` (e.g., Phoenix
controllers' `action/2`). The spec includes macro-generated surfaces in
scope — `realized_by.api_boundary: ["MyController.action/2"]` must resolve
even when `action/2` lives only in the DSL's expansion. A source-first
resolver would emit `branch_guard_dangling_binding` for these, which is a
false positive that erodes trust in the guard.

After compile, beam is authoritative: `function_exported?/3` answers
existence, and `:beam_lib.chunks(beam, [:debug_info])` gives the
post-expansion AST. Source-AST lookup remains useful as a fallback for
pre-compile contexts, modules that were deliberately stripped of debug_info,
and dev-experience messages.

## Decision

`SpecLedEx.Realization.Binding.resolve/2` follows this precedence:

1. Call `Code.ensure_loaded/1` on the target module. If it returns
   `{:error, _}` (module does not exist), proceed to step 3.
2. If `function_exported?(mod, fun, arity)` is true, extract the function's
   AST from `:beam_lib.chunks(beam, [:debug_info])` and return
   `{:ok, ast}`.
3. Fall back to source-AST lookup: locate the source file via the compile
   manifest, parse it, find the `def fun(args)` clause with the right
   arity. Return `{:ok, ast}` on hit.
4. If both paths miss, return `{:error, :not_found}` carrying the declared
   MFA and the search paths consulted.

This rule applies to every tier that resolves MFAs: `api_boundary`,
`implementation`, `expanded_behavior`, `use`, `typespecs`.

## Consequences

- Positive: Phoenix-like macro consumers no longer produce
  `branch_guard_dangling_binding` false positives.
- Positive: beam-first matches the compile model — after compile, beam is
  ground truth.
- Negative: pre-compile or deliberately stripped modules fall through to
  source. Consumers that intentionally strip debug_info (via `@compile
  {:no_debug_info, true}`) will have bindings that resolve inconsistently
  across tiers. Mitigated by `detector_unavailable` at the tier level (see
  expanded_behavior_tier spec).
- Negative: resolution now has I/O cost via `Code.ensure_loaded/1`. In
  practice this is negligible for project-sized module counts but is
  documented.
