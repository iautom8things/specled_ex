# Tag Scanning Shall Read Test Files Via AST, Not Compilation

## Context

Linking requirements to the tests that cover them requires knowing which test blocks
carry `@tag spec: "<id>"`. Two mechanisms were available:

1. **Compilation-based**: Run `mix test --dry-run` or compile each test file and
   introspect module attributes. Captures every form ExUnit accepts but forces the
   host project's full compile graph to succeed, pulls in Application state, and
   couples the scanner to the test-suite lifecycle.
2. **AST-based**: Parse each test file with `Code.string_to_quoted/2` and pattern
   match the supported `@tag`/`@moduletag` shapes on the resulting AST.

The spec tooling runs in contexts where compiling the host app is undesirable: CI
fast-path, `mix spec.check` with test deps unavailable, and agent sessions that
deliberately avoid `app.start` (see `specled.decision.no_app_start`). Compilation-based
scanning would re-introduce the hazards that decision ruled out.

## Decision

Implement the tag scanner (`SpecLedEx.TagScanner`) purely over ASTs produced by
`Code.string_to_quoted/2`. Support exactly four shapes, derived from pinned AST
fixtures generated by the test suite:

- `@tag spec: "id"`
- `@tag [spec: "id", ...other_keys...]`
- `@tag spec: ["id_1", "id_2"]`
- `@moduletag spec: "id"`

`@tag spec:` annotations whose value is not a string literal or list of string literals
(e.g. `@tag spec: @module_attr`) are collected separately and surfaced as a
`tag_dynamic_value_skipped` finding — the scanner never guesses the runtime value.
Parse errors are collected (not swallowed) so the verifier can emit a
`tag_scan_parse_error` finding for each unparseable file.

## Consequences

- The scanner has zero runtime dependency on the host project's test deps. It reads
  and parses source files directly and composes cleanly with the `no_app_start`
  decision.
- New annotation shapes (e.g. compile-time helpers that generate tags) require
  updating the pinned AST fixtures and the scanner's pattern set. Shapes outside the
  supported four are surfaced as findings rather than silently ignored.
- Dynamic tag values and broken files become visible problems instead of invisible
  gaps in coverage.
- Line-level granularity is retained per test, but the scanner tracks data at the
  test-definition level; sub-test or describe-block level tracking is explicitly
  out of scope for v1.
